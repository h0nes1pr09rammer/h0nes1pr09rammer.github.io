设计模式--命令模式

Veröffentlicht am 2017-02-15 | in 设计模式

命令模式

定义

将“请求”封装成对象，以便使用不同的请求、队列或者日志来参数化其他对象。命令模式也支持可撤销的操作。

代码演示

/**
 * 接收者类，知道如何实施与执行一个请求相关的操作，任何类都可能作为一个接收者
 * @author CTWLPC
 *
 */
public class Receiver {
    //真正的命令实现
    public void action(){
        System.out.println("Execute request");
    }
}

/**
 * 抽象命令类，用来声明执行操作的接口
 * @author CTWLPC
 *
 */
public interface ICommand {
    void execute();
}

/**
 * 具体命令类，实现具体命令
 * @author CTWLPC
 *
 */
public class ConcereteCommand implements ICommand{
    //具体命令类包含一个接受者，将这个接受者对象绑定于一个动作
    private Receiver receiver;

    public ConcereteCommand(Receiver receiver) {
        // TODO Auto-generated constructor stub
        this.receiver = receiver;
    }
    //调用接受者相应的action
    public void execute() {
        // TODO Auto-generated method stub
        receiver.action();
    }

}
/**
 * 调度者类
 * @author CTWLPC
 *
 */
public class Invoker {
    private ICommand command;
    //设置命令
    public void setCommand(ICommand command){
        this.command = command;
    }
    //执行命令
    public void executeCommand(){
        command.execute();
    }
}

/**
 * 客户端代码
 * @author CTWLPC
 *
 */
public class Client {
    public static void main(String[] args) {
        Receiver receiver = new Receiver();
        ConcereteCommand command = new ConcereteCommand(receiver);
        Invoker invoker = new Invoker();
        invoker.setCommand(command);
        invoker.executeCommand();
    }
}

优缺点

  • 优点：解除了请求者与实现者之间的耦合，降低了系统的耦合度；对请求排队或记录请求日志，支持撤销操作；可以容易的设计一个组合命令；新命令可以容易的加入到系统中。
  • 缺点：因为针对每一个命令都需要设计一个具体命令类，使用命令模式可能会导致系统有过多的具体命令类。

适用场景

  • 当需要对行为进行记录、撤销、重做等处理时。
  • 系统需要将请求者和接收者解耦，使得调用者和接收者不直接交互。
  • 系统需要在不同时间指定请求、请求排队和执行请求。
  • 系统需要将一组操作组合在一起，即支持宏命令。
